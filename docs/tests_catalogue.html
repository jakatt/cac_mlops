CD Pipeline de déploiement — deploy.yml
Déclenché automatiquement après chaque merge sur main (ou via workflow_dispatch)
#ÉtapeDescription
1 check-changes Détecte si Dockerfiles / requirements.txt / fichiers Python de service ont changé → needs_build=true/false. Calcule aussi RESTART_SERVICES (nginx · gradio · grafana · prometheus · loki · promtail).
2 Build API docker build + push cac-mlops-api:latest vers GHCR — skippé si needs_build=false
3 Build MLflow docker build + push cac-mlops-mlflow:latest vers GHCR — skippé si needs_build=false
4 Build Gradio docker build + push cac-mlops-gradio:latest vers GHCR — skippé si needs_build=false
5 Scan Trivy Scan CVE CRITICAL sur les 4 images — bloque le deploy si vulnérabilité non corrigée détectée
6 SSH VPS → git pull git pull origin main sur le VPS (+ stash des fichiers locaux bind-mountés)
7 docker compose up docker compose pull (si build) + docker compose up -d --remove-orphans
8 Restarts ciblés Restart sélectif des conteneurs modifiés uniquement (nginx · gradio · grafana · prometheus · loki · promtail)
9 prefect deploy --all Resynchronisation des schémas de déploiement Prefect — bloque le deploy si schéma périmé
10 Déclenchement flow update-model-flow si config/model_params.yml modifié (T3) · deploy-vps-flow sinon (T2). Rollback images :rollback si smoke test pré-gate échoue.
PREFECT Tests post-deploy — test-api-flow
Déclenché automatiquement par deploy-vps-flow après chaque déploiement VPS réussi · Exécuté dans le réseau Docker (http://nginx:80)
#Task PrefectCe qui est vérifié
1 test-health GET /health → HTTP 200 — confirme que l'API est démarrée et le modèle chargé
2 test-token POST /token (OAuth2 password grant) → JWT valide retourné
3 test-401-sans-token POST /predict sans en-tête Authorization → HTTP 401 (authentification obligatoire)
4 test-200-avec-token POST /predict avec Bearer JWT → HTTP 200 + prediction · probability · model_version
5 test-whatif-vitesse-90-vs-50 Test de cohérence métier — route départementale, nuit sans éclairage, hors agglomération :
proba(vma=90) > proba(vma=50) (Δ ≈ +0.17)
Garantit que le modèle en production répond de manière cohérente avec la physique des accidents (vitesse maximale autorisée plus élevée → risque de blessure grave plus élevé, toutes choses égales).
6 test-429-rate-limit skippé en CD 22 requêtes consécutives → HTTP 429 déclenché par le rate limiter. Actif uniquement en test manuel (évite de saturer le rate limiter pendant les déploiements automatisés).

Les tests post-deploy s'exécutent à l'intérieur du réseau Docker via http://nginx:80 — ils testent l'API en conditions réelles (load balancer, auth, modèle chargé en mémoire) et non via un mock.
