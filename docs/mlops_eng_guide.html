2 — Deploy direct (phase dev — sans GitHub Actions) 
# Flow Prefect
scp src/flows/mon_flow.py deploy@51.159.187.132:/data/cac_mlops/src/flows/
# Effet immédiat au prochain run Prefect (pas de restart nécessaire)

# Config blueprint
scp config/model_params.yml deploy@51.159.187.132:/data/cac_mlops/config/
# Effet immédiat (bind-mount rw)

# Fichier nginx
scp services/nginx/nginx.conf deploy@51.159.187.132:/data/cac_mlops/services/nginx/nginx.conf
ssh deploy@51.159.187.132 "cd /data/cac_mlops && docker compose restart nginx"

# docker-compose.yml (nouveau service ou nouveau volume)
scp docker-compose.yml deploy@51.159.187.132:/data/cac_mlops/docker-compose.yml
ssh deploy@51.159.187.132 "cd /data/cac_mlops && docker compose up -d --no-recreate"

4 — Les 3 triggers de mise en production
#TriggerDéclencheurPipelineGate promotion
1 Nouvelle data ONISR Prefect cron hebdo (check-new-data, lundi 8h) ETL → validation → train → gate → promote → test-api → Kapsule KPI absolus — promu même en légère régression, tant que ≤1 métrique régresse (≥2 → aucune promotion)
2 Nouveau code MLOps push → PR → merge main (hors modèle) build images (si nécessaire) → VPS prépare (pull code+images, flags) → gate (avant toute interruption) → compose up + smoke test → test-api → Kapsule (si OK) Pas de réentraînement
3 Nouveau blueprint DS push → PR → merge main (config/model_params.yml — guard CI bloque tout mélange code+blueprint) train → si meilleur : gate → promote → test-api → Kapsule ; sinon : email DS KPI absolus + delta F1 > +0.01 vs @Production (même test set)

Détecter trigger 2 vs trigger 3 dans deploy.yml

BLUEPRINT_CHANGED=$(git diff HEAD~1 --name-only | grep -c '^config/model_params\.yml$')
# > 0 → update-model-flow/update-model  (trigger 3)
# = 0 → deploy-vps-flow/deploy-vps      (trigger 2)

Split temporel : make_dataset.py auto-détecte les années dans data/raw/. Dernière année = test set (~55k lignes). Toutes les précédentes = train. Aucune configuration manuelle lors de l'ajout d'une nouvelle année. year_acc supprimé des features.

7 — Ajouter un nouveau flow Prefect
  • Créer src/flows/mon_flow.py avec @flow(name="mon-flow")
  • Ajouter l'entrée dans prefect.yaml :
- name: mon-flow
  entrypoint: src/flows/mon_flow.py:mon_flow
  work_pool:
    name: default-process-pool
  • Re-enregistrer : docker exec -w /app cac_mlops-prefect-worker-1 prefect deploy --all
  • SCP le fichier sur le VPS : scp src/flows/mon_flow.py deploy@VPS:/data/cac_mlops/src/flows/

7 alertes Grafana

Groupe security-monitoring (Prometheus, éval 1 min) :

  • brute-force-401 : > 20 erreurs 401 / 5 min
  • ddos-429 : > 50 erreurs 429 / 5 min
  • ram-critical : RAM disponible < 10%
  • disk-data-critical : /data libre < 15%

Groupe logs-monitoring (Loki, éval 2 min) :

  • prefect-error-logs : ERROR ou CRITICAL dans prefect-worker (5 min)
  • no-champion-log : "Aucun algorithme" dans prefect-worker (10 min) — train_flow terminé sans amélioration @Production
  • drift-critical-log : "CRITICAL drift detected" dans prefect-worker

9 — Trivy — scan CVE

Exécuté dans deploy.yml après chaque build (CRITICAL uniquement, fixées uniquement) :

# Reproduire localement
trivy image --severity CRITICAL --ignore-unfixed \
  --ignorefile .trivyignore ghcr.io/jakatt/cac-mlops-api:latest

Pour ignorer un CVE légitimement non-fixable : ajouter dans .trivyignore.
