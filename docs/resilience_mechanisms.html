Détail

VPS · 4 gates — 1 manuelle — 1 rollback — 1 interruption
K8s · 1 gate — 1 rollback — 0 interruption

  • Gate 1 Source data.gouv inaccessible ou fichier absent → flow Prefect stoppé + alerte
  • Gate 2 Pandera CRITICAL : format, colonnes, volumétrie hors norme → ETL stoppé + alerte
  • Gate 3 KPI gate failed : aucun champion RF/XGB/LGBM → @Production inchangé + alerte
  • Smoke test pré-gate Santé VPS vérifiée (90 s sur /health) avant ouverture de la gate : si fail → flow stoppé immédiatement, rien de déployé.
  • Gate manuelle Flow deploy-vps-flow mis en pause : métriques du champion (f1 / recall / auc / acc) affichées dans le Cockpit ou Prefect UI. Rien n'est appliqué en prod à ce stade. GO → promotion @Production + redémarrage API (~30 s). STOP → run annulé, @Production inchangé, aucun impact.
  • Gate 4 VPS test-api (6 tâches) : si fail → rollback alias MLflow @Production + restart API + alerte
  • Gate K8s (si VPS OK) Readiness probe /health : nouveau pod Ready avant suppression de l'ancien → 0 interruption. Si fail → kubectl rollout undo + rollback alias MLflow @Production + alerte

VPS · 4 gates — 1 manuelle — 1 rollback — 1 interruption
K8s · 1 gate — 1 rollback — 0 interruption

  • Gate 1 CI lint/pytest fail → merge bloqué par branch protection + notification GitHub Actions
  • Gate 2 Trivy CVE CRITICAL → deploy annulé avant tout contact VPS + annotation erreur
  • Smoke test pré-gate Santé VPS vérifiée (90 s sur /health) : si fail → flow stoppé, images déjà pull-ées mais aucun conteneur touché.
  • Gate manuelle Flow deploy-vps-flow mis en pause : SHA, images à builder et services à redémarrer affichés dans le Cockpit ou Prefect UI. Images pull-ées mais pas encore appliquées. GO → docker compose up -d. STOP → run annulé, aucun conteneur touché.
  • Gate 3 VPS Smoke test (90 s) après docker compose up -d : si fail → rollback Docker :rollback + alerte
  • Gate 4 VPS test-api (6 tâches) : si fail → rollback Docker :rollback + alerte
  • Gate K8s (si VPS OK) Readiness probe /health : nouveau pod Ready avant suppression de l'ancien → 0 interruption. Si fail → kubectl rollout undo (api / gradio-public / nginx / caddy) + alerte

VPS · 3 gates — 1 manuelle — 1 rollback — 1 interruption
K8s · 1 gate — 1 rollback — 0 interruption

  • Gate 1 CI lint/pytest fail — blueprint pur, pas de build Docker, pas de Trivy
  • Gate 2 KPI gate failed : aucun champion → @Production inchangé + alerte
  • Smoke test pré-gate Santé VPS vérifiée (90 s sur /health) : si fail → flow stoppé, rien de déployé.
  • Gate manuelle Flow deploy-vps-flow mis en pause : métriques du nouveau blueprint affichées dans le Cockpit ou Prefect UI. Rien n'est appliqué en prod à ce stade. GO → promotion @Production + restart API (~30 s). STOP → run annulé, aucun impact.
  • Gate 3 VPS test-api (6 tâches) : si fail → rollback alias MLflow @Production + alerte
  • Gate K8s (si VPS OK) Readiness probe /health (pod avec nouveau modèle) : nouveau pod Ready avant suppression de l'ancien → 0 interruption. Si fail → kubectl rollout undo + rollback alias MLflow @Production + alerte

Mécanismes spécifiques K8s (Kapsule)

Rolling update : le nouveau pod doit passer la readiness probe (/health) avant que l'ancien soit supprimé. Aucune requête n'est perdue — interruptions de service à 0 pour les 3 triggers. Rollback = kubectl rollout undo (vs :rollback Docker côté VPS).

Double protection : Kapsule n'a pas de gate manuelle propre. Le verrou humain est côté VPS (deploy-vps-flow mis en pause). deploy-kapsule-flow ne démarre qu'après GO et test-api VPS réussi — K8s est protégé par deux verrous en série : gate humaine + validation VPS.

Alerte = log Loki → alerte Grafana → email SMTP · Exception : notification GitHub Actions pour les gates CI (T2 Gate 1–2 · T3 Gate 1 uniquement — pas de build, pas de Trivy en T3) Rollback VPS T1/T3 : alias MLflow @Production · Rollback VPS T2 : images Docker :rollback · Rollback K8s (tous triggers) : kubectl rollout undo
