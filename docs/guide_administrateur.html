1 — Contexte & Objectif
Problème métier & cible

Prédire la gravité d'un accident de la route à partir de ses caractéristiques au moment de la déclaration : conditions météo, type de voie, heure, caractéristiques de l'usager et du véhicule.

grav = 1  →  victime avec blessure grave ou décès  (PRIORITAIRE)
grav = 0  →  blessé léger ou indemne               (NON PRIORITAIRE)
KPIs & seuils de performance
KPISeuil minimumPourquoi
F1-score≥ 0.60Équilibre précision/rappel sur classes déséquilibrées
AUC-ROC≥ 0.77Capacité discriminante globale
Accuracy≥ 0.72Indicateur de référence global
Recall (grav=1)≥ 0.58Minimiser les faux négatifs — ne pas manquer un blessé grave
Modèle actuel @Production : lgbm_accidents — cumul 2021+2022+2023 · F1=0.664 · AUC=0.839 · Recall=0.631 · Acc=0.783 · Marge ~8% pour absorber la variance inter-années

Seuil delta minimal : +0.01 sur F1 pour remplacer @Production (évite les swaps sur bruit statistique).

Seuil de régression : si le nouveau modèle régresse vs @Production sur ≥2 métriques → aucune promotion, @Production inchangé.

2 — Source des données & stratégie d'ingestion
Source : ONISR (Observatoire National Interministériel de la Sécurité Routière) — data.gouv.fr · Dataset ID : 53698f4ca3a729239d2036df · Publication annuelle (données N publiées en mai-juin N+1)
Périmètre retenu : 2021 → 2024

Schéma ONISR stable depuis 2021 (refonte majeure en 2019). Données 2021→2023 = entraînement cumulatif. Données 2024 = simulation production drift.

Année calendaireEntraînement cumulatifDrift de features
2023 (1ère prod)2021 → Modèle v1 @Production— (pas de référence)
20242021+2022 → Modèle v22022 vs réf. 2021
20252021+2022+2023 → Modèle v32023 vs réf. 2021+2022
20262021+2022+2023+2024 → v42024 vs réf. 2021+2022+2023

Chaque nouvelle année sert à la fois de test set temporel (évaluation F1/AUC) ET de sujet du drift de features vs années précédentes — les deux opérations sont indépendantes.

3 — Pipeline ETL détaillé
ÉtapeScriptSortieSi KO
1. Ingestionimport_raw_data.pydata/raw/{year}/*.csv — nom canonique {catégorie}-{year}.csv imposé au download (indépendant du nom serveur)—

6. Versioning DVCDVC + gitS3 — .dvc commités (raw et preprocessé, versionnés séparément)—
Preprocessing détaillé (make_dataset.py)
  • Feature engineering : âge victime (year_acc - an_nais), heure (hrmn), nb_victim, nb_vehicules
  • Nettoyage : remplacement -1 et 0 → NaN · recodage catv (40→7 classes) · atm (9→2) · Corse 2A→201, 2B→202
  • Fusion 4 tables sur Num_Acc · tri par grav DESC · dédoublonnage (1 ligne/accident — victime la plus grave)
  • Cible binaire : grav {2,3,4} → {0,1,1}
  • Suppression colonnes identifiants (id_usager, year_acc)
  • Train/Test split : 70/30, random_state=42
  • Imputation NaN sur 4 colonnes par mode(X_train)
Schéma détaillé — flux ETL ↔ DVC

Deux artefacts distincts sont versionnés dans DVC, à deux moments différents du pipeline — le schéma ci-dessous montre lequel, et surtout ce qui est réellement corrigé à chaque étape.

download_task import_raw_data.download_year() → data/raw/{year}/{catégorie}-{year}.csv — nom canonique imposé, indépendant du nom serveur
↓
validate_task load_and_validate_year(year) — N1 Format · N2 Schéma + auto-correction (known_fixes + Pandera coerce, en mémoire) · N3 Qualité
❌ CRITICAL → STOP · rien versionné dans DVC
✅ OK / ⚠️ WARNING / 🔧 AUTO_CORRECTED — toujours en mémoire à ce stade
dvc_push_task dvc add + push data/raw/{year}, puis commit data/raw/{year}.dvc → origin/main [skip ci] — le tout dans un clone git jetable (PAT récupéré depuis S3, créé/détruit à chaque exécution — indépendant de /app/.git, jamais présent dans l'image)
→
DVC #1 — Raw CSV tel que téléchargé, intact. Les corrections restent en mémoire — jamais réécrites sur disque.
↓
years = toutes les années disponibles (cas standard, cumul=true) — pas seulement la nouvelle
↓
preprocess_task make_dataset.process_years(years) — pour chaque année du cumul : load_and_validate_year rejoué (y compris les années déjà en prod) + feature engineering
❌ CRITICAL sur une seule année du cumul → tout le preprocessing s'arrête
↓ features finales, corrections désormais persistées
dvc_push_preprocessed_task dvc add + push data/preprocessed/cumul_{years}, puis commit data/preprocessed/cumul_{years}.dvc → origin/main [skip ci] — même clone jetable
→
DVC #2 — Preprocessed X_train/X_test/y_*.csv — seul endroit où renommages et coercion sont réellement persistés.

Le raw versionné dans DVC reste une archive fidèle de ce qu'ONISR a publié (nom de fichier mis à part) — jamais « nettoyé ». Toute correction de contenu ne devient visible que dans l'artefact preprocessé : DVC #1 = source brute immuable, DVC #2 = dérivé recalculable.

StadeVersionné (.dvc + S3)Contenu réel
Après validate (N1/N2/N3 OK)data/raw/{year}.dvcCSV brut, nom canonique — contenu jamais corrigé sur disque
Après preprocessdata/preprocessed/cumul_{years}.dvcX_train/X_test/y_*.csv — seules données réellement corrigées et persistées
4 — Validation de schéma — Pandera

Pandera est retenu (léger, natif pandas, intégrable Prefect, unitestable) vs Great Expectations (overkill) et pur Python (non standardisé).

NiveauVérifiéSi KO
1 — Format fichierSéparateur ( ; ) · encodage (UTF-8) · 4 fichiers présents · aucun vide❌ CRITICAL → pipeline stoppé, modèle N-1 actif
2 — Schéma colonnesColonnes requises présentes · types corrects · colonnes inconnues (nouvelles ONISR)❌ CRITICAL (colonnes requises) / ⚠️ WARNING (inconnues)
3 — Qualité donnéesVolume 40k–90k accidents · codes modalités connus · taux NaN <30% · lat/long France⚠️ WARNING → log + pipeline continue
Auto-correction — au-delà de la détection

En plus de détecter et signaler les anomalies, certaines sont désormais corrigées automatiquement avant que la donnée n'atteigne le preprocessing — tracé dans le rapport sous un niveau dédié AUTO_CORRECTED (informatif, n'escalade jamais au-dessus de OK/WARNING).

MécanismeCorrigeExemple
Registre centralisé
known_fixes.py		Renommages ONISR connus + espaces insécables/superflus sur colonnes texteAccident_Id → Num_Acc (ONISR 2022+) — appliqué identiquement en validation et en preprocessing (source unique, ex-duplication corrigée)
Coercion de type
Pandera coerce=True		Colonne lue avec le mauvais type à cause d'un artefact de formatageColonne entière lue en texte (espace insécable résiduel) → reconvertie en int. Échec de conversion (valeur réellement invalide) → jamais de NaN silencieux, redescend proprement en WARNING
Fallback fuzzy nom de fichier
import_raw_data.discover_raw_files		Nom de fichier source légèrement différent de l'attenduIncident réel carcteristiques-2021.csv (faute de frappe ONISR, jamais rattrapée) — similarité de chaîne (difflib) retrouve la bonne catégorie, WARNING loggué pour traçabilité

Depuis l'unification du pipeline (load_and_validate_year), make_dataset.py ne relit plus les CSV séparément : il consomme directement les DataFrames déjà corrigés/validés par schema_validator.py. Conséquence utile : chaque cycle ETL revalide automatiquement toutes les années du cumul avec les règles les plus récentes, pas seulement la nouvelle — un correctif qui casserait silencieusement une année déjà en production est détecté au prochain cycle, sans tâche dédiée.
