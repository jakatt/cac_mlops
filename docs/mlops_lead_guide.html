2 — Les 3 scénarios de mise en production
Trigger 1 — Nouvelle data ONISR (automatique, hebdo)

check-new-data-flow tourne chaque lundi 8h UTC. S'il détecte une nouvelle année sur data.gouv.fr (4/4 fichiers disponibles) :

ETL → validation schéma → train (année incluse, test set temporel) → sélection champion
  → GATE MANUELLE → promote → test-api (6 tests) → Kapsule (si OK) → drift

Règle de promotion T1 : gate KPI absolue (F1≥0.60, AUC≥0.77, Recall≥0.58, Acc≥0.72). Le nouveau modèle est promu même s'il régresse légèrement vs @Production (test sets différents). Tolérance : régression sur au plus 1 métrique. Régression sur ≥2 → @Production inchangé.

Ton rôle : valider la gate — onglet Cockpit du cockpit Gradio (bouton GO/STOP) ou Prefect UI — quand tu reçois la notification email.

Trigger 2 — Nouveau code MLOps 
build images (si nécessaire) → git pull VPS + pull images (préparation, sans interruption)
  → GATE MANUELLE (avant toute interruption VPS) → docker compose up + redémarrages ciblés
  → smoke test → test-api (6 tests) → Kapsule (si OK)

La gate intervient avant que quoi que ce soit soit appliqué sur le VPS — rien n'est interrompu tant que tu n'as pas validé. Un STOP annule proprement sans rollback (rien n'a encore tourné).

Ton rôle : valider la gate (onglet Cockpit — SHA du commit, images à reconstruire, services à redémarrer affichés) ou Prefect UI.

Trigger 3 — Nouveau modèle DS 
extract blueprint → train avec nouveaux params → compare @Production
  → si meilleur (+0.01 F1, même test set) : GATE MANUELLE → promote → test-api → Kapsule
  → si non meilleur : @Production inchangé, email notification DS

Règle de promotion T3 : KPI absolus + delta F1 > +0.01 vs @Production. La comparaison est valide car les deux modèles utilisent le même test set (même année).

Ton rôle : valider la gate (onglet Cockpit ou Prefect UI) si un champion est trouvé. Si le merge inclut aussi du code, la gate affiche le SHA + services — tout est appliqué seulement après validation.

3 — Valider une gate manuelle
Via Gradio Cockpit — onglet Cockpit (recommandé)
  • Onglet Cockpit → la file d'attente liste les déploiements deploy-vps-flow en pause
  • Sélectionner le run → le détail affiche les métriques du champion (F1/Recall/AUC vs @Production) ou le SHA/services impactés
  • GO → applique le déploiement (promote et/ou compose up selon le trigger) → test-api → Kapsule (si OK)
  • STOP → annule proprement — rien n'a encore été appliqué, aucun rollback nécessaire
Via Prefect UI (alternative)
  • Ouvrir Prefect UI → Flow Runs
  • Trouver le run deploy-vps-flow ou update-model-flow en état Paused
  • Vérifier les métriques dans les logs (F1, Recall, AUC du champion, ou SHA/flags si code)
  • Cliquer Resume pour valider → promote et/ou compose up → test-api → Kapsule (si OK)
  • En cas de doute : laisser le run expirer (24h timeout) ou cliquer Cancel

5 — Drift — surveillance et action

Le drift est calculé automatiquement après chaque cycle de retrain. Pour le vérifier manuellement :

Gradio Cockpit → onglet Cockpit → accordéon Drift → sélectionner l'année

Prefect UI : Deployments → drift-check → Quick run

Drift shareNiveauAction
< 10%OKRien
10–25%WARNINGEmail d'alerte — surveiller l'évolution
> 25%CRITICALEmail d'alerte — planifier le prochain cycle annuel manuellement

Aucun réentraînement automatique sur drift : les labels N+1 sont indisponibles (ONISR publie avec ~2 ans de délai). Réentraîner sur les mêmes données produirait un modèle identique. Le drift est un signal pour décider quand déclencher le prochain cycle via check-new-data-flow.

6 — Promouvoir ou rollback un modèle
Via Gradio Cockpit (recommandé)

Onglet Cockpit → accordéon Modèles → sélectionner la version → Promouvoir @Production

Puis : SSH → docker compose restart api pour charger la nouvelle version.

Via MLflow UI

http://100.117.99.62:5001/#/models → sélectionner le modèle → Aliases → éditer @Production.

Via CLI Python 
python3 - <<'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5001")   # tunnel SSH actif
client = mlflow.tracking.MlflowClient()

# Lister les versions
for v in client.search_model_versions("name='lgbm_accidents'"):
    print(f"v{v.version}  run_id={v.run_id[:8]}")

# Promouvoir (MLflow v3 — aliases, pas stages)
client.set_registered_model_alias("lgbm_accidents", "Production", "7")
EOF
