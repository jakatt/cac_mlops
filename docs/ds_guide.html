Guide Data Scientist

Développer et optimiser un modèle en local, tracer les expériences dans MLflow, soumettre un blueprint via PR. La mise en production est déclenchée automatiquement après merge (trigger 3).

1 — Setup initial (une seule fois) 
git clone git@github.com:jakatt/cac_mlops.git
cd cac_mlops

python3 -m venv my_env
source my_env/bin/activate
pip install -r requirements.txt
pip install "dvc[s3]>=3.0"
pip install -e .
cp .env.example .env

Credentials DVC (accès données brutes sur Scaleway S3)

dvc remote modify --local scaleway access_key_id     <ACCESS_KEY>
dvc remote modify --local scaleway secret_access_key <SECRET_KEY>

Ces credentials vont dans .dvc/config.local — gitignored, ne jamais commiter.

.env est chargé automatiquement par toute commande python -m src.xxx (via python-dotenv) — pas besoin d'exporter ses variables à la main dans le shell.

Tailscale (accès direct au VPS — MLflow, Prefect...)

Installer Tailscale et rejoindre le tailnet du projet (demander l'accès au MLOps lead). Une fois connecté, tous les services du VPS sont joignables directement via son IP Tailscale (100.117.99.62) — aucun tunnel SSH nécessaire.

2 — Récupérer les données

Recommandé — une seule commande : synchronise la branche DS avec origin/main, puis dvc pull les données brutes et le dataset preprocessé cumulatif (le génère localement en secours si l'ETL ne l'a pas encore versionné) :

./scripts/ds_session_start.sh

Détail de ce que fait le script, si tu préfères le faire à la main ou comprendre les étapes :

3 — Tracer les expériences dans MLflow

Les runs locaux remontent sur le MLflow partagé du VPS — directement via Tailscale (MLFLOW_TRACKING_URI=http://100.117.99.62:5001 déjà dans .env), aucun tunnel SSH nécessaire (cf. section 1). MLflow UI : http://100.117.99.62:5001.

Levier 2 — Approche 1 : Override CLI (rapide, 1 run)

Objectif : identifier les valeurs optimales à consigner ensuite dans config/model_params.yml.

# MLFLOW_RUN_MODE=explore (défaut) → runs dans "accidents_severity_dev"
python -m src.models.train_model --year 2023 --cumul --algorithm lgbm \
  --n-estimators 500 --num-leaves 127 --learning-rate 0.03 --no-register

Tous les hyperparamètres du blueprint sont overridables en CLI (pas seulement 4) — la liste complète par algo est en section 5. python -m src.models.train_model --help affiche tous les flags disponibles. --no-register évite d'enregistrer le modèle dans le Model Registry pour un run purement exploratoire.

Levier 2 — Approche 2 : Grid Search / RandomizedSearch / Optuna (exhaustif) 
from sklearn.model_selection import RandomizedSearchCV
from lightgbm import LGBMClassifier
import pandas as pd, numpy as np

X_train = pd.read_csv("data/preprocessed/cumul_2021_2022_2023/X_train.csv")
y_train = np.ravel(pd.read_csv("data/preprocessed/cumul_2021_2022_2023/y_train.csv"))

param_grid = {
    "n_estimators":      [200, 300, 500],
    "num_leaves":        [31, 63, 127],
    "learning_rate":     [0.01, 0.05, 0.1],
    "min_child_samples": [10, 20, 50],
    "colsample_bytree":  [0.8, 1.0],
}
clf = LGBMClassifier(n_jobs=-1, random_state=42, verbose=-1)
search = RandomizedSearchCV(clf, param_grid, n_iter=20, scoring="f1",
                             cv=3, random_state=42, n_jobs=-1)
search.fit(X_train, y_train)
print(search.best_params_)
# → copier ces valeurs dans config/model_params.yml

Les paramètres infra (n_jobs=-1, random_state=42, verbose=-1) sont gérés automatiquement — ne pas les inclure dans la grille.

Comparer un run à la prod

Chaque training affiche automatiquement, dans le terminal, un tableau comparant les 4 métriques (accuracy/f1/auc/recall) à celles du modèle actuellement @Production, avec un verdict Send to prod ? (oui/non) — plus besoin d'aller chercher les métriques de la prod ailleurs. Si "OUI", la commande pour promouvoir ce run est directement affichée (cf. section 4).

Comparer plusieurs runs d'exploration entre eux

MLflow UI → Experiments → accidents_severity_dev → trier par F1 / AUC / Recall. Utile pour choisir entre plusieurs de tes propres essais — la comparaison face à la prod, elle, est automatique (ci-dessus).

Pour repartir sur une base d'exploration propre (vider tes anciens runs) : ne jamais supprimer l'expérience elle-même dans l'UI (bloquerait tout training suivant avec une erreur MLflow tant qu'elle n'est pas restaurée/purgée). Utiliser à la place :

MLFLOW_CLEANUP_EXPERIMENT=accidents_severity_dev MLFLOW_CLEANUP_KEEP=0 \
  python -m src.scripts.mlflow_cleanup
4 — Soumettre un blueprint vers la production (trigger 3)

Quand le tableau affiché en fin de training indique "Send to prod ? OUI" :

Étape 1 — Générer le blueprint depuis le run champion

Copier la commande affichée juste en dessous du verdict (le run_id est déjà rempli) :

python -m src.scripts.extract_blueprint <run_id>

Cette commande tague le run export_to_prod=true dans MLflow (traçabilité) et écrit directement config/model_params.yml — une seule action, pas besoin de passer par l'UI MLflow ni d'éditer le YAML à la main. Ajouter --dry-run pour prévisualiser sans écrire.

Étape 2 — Commiter et ouvrir une PR 
git checkout DS   # branche de travail DS habituelle
git add config/model_params.yml
git commit -m "blueprint: nouveaux hyperparamètres lgbm"
git push origin DS

Ouvre une PR vers main. Le MLOps lead review et merge.

⚠️ Important : ne pas mélanger modification de config/model_params.yml et changements de code source (src/, services/) dans la même PR — le CI bloquera avec un message d'erreur explicite. Si les deux sont nécessaires, ouvrir deux PRs séparées : PR1 code, PR2 blueprint.
Ce qui se passe automatiquement après merge

deploy.yml détecte que config/model_params.yml a changé et déclenche update-model-flow qui :

  • Entraîne les 3 algos avec les hyperparamètres de config/model_params.yml sur données prod
  • Meilleur → gate manuelle (onglet Cockpit ou Prefect UI) + promote @Production + restart API
  • Pas meilleur → email de notification (stop, aucune promotion)

8 — Branches et workflow Git 
DS  →  PR vers main  →  review MLOps lead  →  merge  →  CI + deploy auto
  • Travailler sur la branche DS, jamais directement sur main
  • Resynchroniser DS sur origin/main avant de commencer une nouvelle série de changements (./scripts/ds_session_start.sh le fait automatiquement, cf. section 2)
  • Les pushes sur main déclenchent deploy.yml (build images + deploy VPS)
  • Toujours ouvrir une PR — les workflows CI vérifient les tests avant merge
