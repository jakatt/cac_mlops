Différence fondamentale vs VPS

Le rolling update Kubernetes remplace les pods un par un : le nouveau pod doit passer la readiness probe (/health) avant que l'ancien soit supprimé. Aucune requête n'est perdue — les interruptions de service tombent à 0 pour les 3 triggers. Le rollback passe de images Docker :rollback à kubectl rollout undo.

4 gates — 1 rollback — 0 interruption

  • Gate 1 Vérification source data.gouv : URL inaccessible ou fichier absent → flow Prefect stoppé + alerte, Kapsule jamais touché
  • Gate 2 Pandera CRITICAL : format, colonnes, volumétrie hors norme → ETL stoppé + alerte email SMTP
  • Gate 3 KPI gate failed : aucun champion → @Production inchangé, Kapsule non mis à jour + alerte Grafana
  • Gate 4 Readiness probe K8s (/health) : nouveau pod Ready avant suppression de l'ancien → 0 interruption ; si fail → kubectl rollout undo + rollback alias MLflow @Production

4 gates — 1 rollback — 0 interruption

  • Gate 1 CI lint/pytest fail → merge bloqué + notification GitHub Actions
  • Gate 2 Trivy CVE CRITICAL → deploy annulé avant tout contact VPS/Kapsule
  • Gate 3 Smoke test VPS deploy.yml (120 s) : si fail → Docker VPS :rollback + docker compose up -d, Kapsule non déclenché
  • Gate 4 Readiness probe K8s (/health) : nouveau pod Ready avant suppression de l'ancien → 0 interruption ; si fail → kubectl rollout undo deployment/api — si Prefect VPS test-api KO avant K8s : Docker VPS :rollback + Kapsule non déclenché

4 gates — 1 rollback — 0 interruption

  • Gate 2 Trivy CVE CRITICAL : identique T2
