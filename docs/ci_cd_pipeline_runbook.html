MLOps · Accidents routiers — CI/CD post-merge

Runbook — le pipeline post-merge a échoué

Que faire quand le bandeau statut de l'onglet Cockpit (ou l'alerte email Grafana CI/CD — pipeline deploy.yml échoué) signale un échec. Le code mergé sur main et les images poussées sur GHCR sont déjà sains dans tous les cas ci-dessous — seule la propagation vers le VPS/Kapsule a été interrompue. Refaire une PR n'est jamais nécessaire.

Stage : github_cd

Scan Trivy — CVE CRITICAL détectée bloquant
SymptômeJob build en échec sur GitHub Actions, annotation ::error::CVE CRITICAL détectée — deploy bloqué.
CauseUne image (api/gradio/mlflow/caddy) contient une vulnérabilité CRITICAL non ignorée par .trivyignore.
Résolution
  • Lire le rapport Trivy dans les logs du job build pour identifier le paquet concerné.
  • Mettre à jour la dépendance/l'image de base si un correctif existe, ou l'ajouter à .trivyignore si le risque est jugé acceptable (documenter pourquoi).
  • Re-pousser un commit — le pipeline repart normalement, aucune action manuelle de déploiement nécessaire.
prefect deploy --all échoué (resync des schémas) bloquant
SymptômeJob trigger-deploy en échec, annotation ::error::prefect deploy --all échoué.
CauseGénéralement transitoire — prefect-worker pas encore prêt, ou serveur Prefect momentanément indisponible.
Résolution
  • Onglet Actions GitHub → le run concerné → Re-run failed jobs (ou gh run rerun <run_id> --failed). Aucun rebuild d'image, quasi instantané.
  • Si ça persiste : SSH sur le VPS, docker compose ps prefect-worker pour vérifier qu'il tourne, puis rejouer manuellement docker compose exec -T prefect-worker prefect deploy --all.

Stage : prefect_trigger

Échec de prefect deployment run (aucun flow run créé) bloquant
SymptômeLigne "Déclenchement Prefect" du bandeau Cockpit au rouge — aucun run n'apparaît dans la file d'attente ni dans l'onglet Orchestration.
CauseLe plus souvent un schéma de paramètres périmé, ou un incident transitoire réseau/Prefect au moment du déclenchement.
Résolution
  • Le plus rapide : bouton « Réessayer le déclenchement » dans l'onglet Cockpit — rejoue automatiquement prefect deployment run avec exactement les mêmes paramètres (SHA, needs_build, restart_services) que la tentative échouée.
  • Si le bouton indique des informations insuffisantes : relancer le job trigger-deploy depuis GitHub Actions (Re-run failed jobs).
  • En dernier recours : SSH sur le VPS et rejouer manuellement la commande prefect deployment run '<flow>/<deployment>' --param ... (voir .github/workflows/deploy.yml pour la syntaxe exacte selon T2/T3).

Le flow run lui-même (gate manuelle, promotion, rolling update Kapsule), une fois créé avec succès, a sa propre gestion d'erreur — alertes event=alert déjà couvertes par les règles Grafana existantes (mlops-alert-critical/mlops-alert-warning) et visibles dans l'onglet Orchestration (« Derniers flows exécutés »). Ce runbook couvre uniquement la fenêtre avant la création du flow run.
